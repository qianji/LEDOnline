Objects

An LED datum is a rational number, an atom, a finite set of data, a tuple of data, or  a lambda expression.

Rational numbers are written in LED as numerals. A numeral  consists of one or more digits, possibly followed is a decimal point and one or more additional digits. For example, 21.221  is a numeral.

A truth value is either truth or falsehood, written in LED as true and false, respectively.

A atom consists of a backquote, followed by a letter, followed by zero or more letters and digits. For example, `foobar2 is a atom. LED is case sensitive, so for example `foo and `Foo are different atoms.

A lambda expression consists of the identifier lambda, followed by one or more variables, a period (.),  and then a term. For example,  lambda i j. i+(2*j) is a lambda expression.

A roster set can be written as zero or more terms, separated by commas and enclosed in braces. For example, {1,2,3} is a set. As usual{x1,...,xn} is set whose members are x1,...,xn.

An n-tuple, where n ≥ 2, is written as n terms, separated by commas and enclosed in parentheses. For example, (1,2,3) is a 3-tuple. As usual, (x1,...,xn) is the n-tuple whose i’th coordinate is xi  for 1 ≤ i ≤ n. For any n ≥ 2, an n-tuple of objects may be called a tuple.

These are the only data objects in LED. Certain objects may be written using shorthand notations described below.

A sequence is a set of pairs whose distinct first elements are  {1,...,n} for some n ≥ 0. As a shorthand, the sequence {(1,x1), ...,(n,xn)} may be written as <x1, ...,xn>. For example, the sequence  {(1, `foo), (2,`bar)} can be written as <`foo,`bar>.

A string character is either a printable ascii character which is not backslash or double-quote,  or one of the following two-character special string atoms:

 \\    \t    \r   \" 

A quoted string consists of zero or more string characters enclosed in double quotes ( "). A quoted string can be written in LED as shorthand for a sequence of integers, whose respective members are the ascii codes represented by its string atoms (where the special string atoms represent ascii codes for backslash, horizontal-tab, carriage-return, and double-quote), For example, the quoted string "hi " can be written for the sequence <104,105>.

Notes on semantics

  1. x/y is defined if and only if  y≠0.
  2. m mod n is not defined when n ≤ 0
  3. 0^0 is not defined
  4. false & q is false even if q is undefined, and false => q is true even when q is undefined. The other Boolean connectives are strict, in the sense that all of their arguments must be defined in order for them to return a value.
  5. Rational numbers, sets, tuples, atoms, truth values, and lambda expressions are pairwise disjoint; that is, d=e is false if d is one of these and e is another. Equality of lambda expressions is intensional; that is, two lambda expressions are equal if and only if one can be obtained from the other by alpha reduction (i.e., consistent renaming of variables).
  6. In all quantifiers and aggregates, P must be a simply solvable sentence, defined below.

Simply solvable sentences

In all quantifiers and aggregates shown in the table, P must be a simply solvable sentence. Intuitively, a sentences is “simply solvable” if its free variables are restricted to finite sets in a straightforward way. This is defined more precisely below.

A pattern is an expression in which no variable occurs free in an argument of a function. More precisely a pattern is either a closed expression, a variable, a tuple (x1,...,xn) where each xi is a pattern, or a set roster {x1,...,xn} where each xi is a pattern. The set of simply solvable sentences is the smallest set of sentences satisfying the following:

  • Every closed sentence is simply solvable.
  • Any atomic sentence, all of whose terms are either constants or variables ranging over a finite type, is simply solvable.
  • x in s, where x is a pattern and s is closed object term, is simply solvable.
  • x = y is simply solvable if at both are patterns and at least one of the two is closed.
  • P & Q is simply solvable if P is solvable and Q’ is simply solvable, where Q’ is obtained by from Q by replacing the free variables of P with arbitrary constants.
  • P or Q is simply solvable if P and Q are both simply solvable.
  • P => Q is simply solvable if ~(P & ~Q) is simply solvable.
  • P <=> Q s simply solvable if both P and Q are simply solvable.

For example, the following sentence is simply solvable:

x in {1..10} & x mod 2 = 0 & y in {1..x} & x+y<9

and the following is not:

     a,b,c,n in Int & n>2 & a^n + b^n = c^n

LED Programs

An untyped LED program is an asciii text file which consists of documentary text, variable declarations, and function definitions.

Within an LED program, a start symbol is a greater-than sign (>) that is the first non-whitesapce character of a new line, Each formal element of the program, i.e. definition or declaration, begins with start symbol and proceeds to the next symbol atom or blank line. All other text within the program is documentary. The following is an example of documented simple type definitions in  an LED program.

A variable is simply an identifier. An identifier occurs as a variable if (1) it occurs in a place where only a variable may occur (function parameter, quantified variable, in a lambda expression, etc.), or (2) it is declared as a variable before it is used in the program. Variable declarations are written as follows:

        var x

        vars x1,..., xn

where the x’s are identifiers. A variant of an identifier x is either x itself, or x followed by 1 or more digits. Any variant of a variable declared in a program is a variable of that program.

The general form of an LED function definition is

if P then f(x1,…,xn) =  B1 if G1;…; Bk if Gk

where

  • f  is a function symbol,  B’s are terms, G's are statements,  x’s are variables,  P[x1:=c1,…,xn:=c1] is a simply solvable sentence for any substitution [x1:=c1,…,xn:=c1], and
  • no variable occurs free in the right hand side (after =)  that does not occur free in the left hand side (before =).

The reading of the definition is:  

f is a partial function from T1 to T2, and for  i in {1..k}  if P and Gi  hold then f(x1,..xn) is defined to be Bi

The following syntactic sugar can be used when writing function definitions:

  • If P is true then  if P  can be omitted.
  • We may write  Bk otherwise  in place of  Bk if ~(Gk or … or Gk-1)
  • We may write G in place of  G if true, when the latter is the entire body of the definition.
  • We may write part of the precondition P at the end of the definition using the keyword where. That is, we may write

         If P then f(x1,...,xn) =  B1 if G1 ; … ; Bk if Gk  where P’

in place of

          If P & P’ then f(x1,...,xn) =  B1 if G1 ; … ; Bk if Gk

  • In case a function f ranges over truth values, we may write

           if P then f(x1,…,xn) iff Q

in place of

         if P then f(x1,…,xn) =  Q
